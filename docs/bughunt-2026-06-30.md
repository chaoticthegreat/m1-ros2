## Full detail

### BUG 01 — [HIGH] Slow target motion (<6 mm/s) stops being tracked: steady-reuse branch never re-solves, held-sentinel skips the polish, fingertip lags target unboundedly while _dist falsely reports ~0
- **File:** `ros2_ws/src/m1_control/m1_control/kinematics.py` lines 844-907 (category: correctness, finder: kin-ik, verifier confidence: high)
- **Problem:** solve_step's steady-state reuse branch (the `elif match and jump < _IK_TARGET_EPS and ... max(cache["dist"]) <= _IK_REACQUIRE_POS` at lines 850-854) compares the target jump against the PREVIOUS tick's cached target, and `store()` (lines 836-842, called every tick) refreshes that cached target snapshot to the current value. So a target that creeps by less than `_IK_TARGET_EPS` (1e-4 m) per tick is treated as static FOREVER: the branch reuses the stale cached `q_best` / `cache["dist"]` and never re-solves toward the moving goal. Worse, because the reused `q_best` equals the measured config, `dq = q_best - q_meas_vec` is ~0 so `nrm < IK_CMD_DEADBAND`, and the reused `solved_resid = max(dist_best) = cache["dist"]` is the tiny STALE residual (<= _IK_REACQUIRE_POS), so the held-sentinel early-return at lines 906-907 fires and the per-arm Cartesian polish (lines 912-935) — the safety net that would otherwise drive the fingertip onto the current target — is entirely skipped. Net effect: at sustained target speeds below 6.0 mm/s (= _IK_TARGET_EPS * 60 Hz) the fingertip falls behind the target without bound, and the returned `_dist` falsely claims the arm is on target. This is exactly the precision/slow-teleop regime (e.g. Quest precision mode at 0.25x scale, or a slow deliberate hand move).
- **Evidence:** Ran /tmp/repro_drift3.py: a left-arm target swept along +x at constant speed for 10 s after settling. At 1.0 mm/s (step 16.7 um/tick) max fingertip error = 9.995 mm; at 3.0 mm/s = 29.995 mm; at 5.0 mm/s = 49.995 mm; at 6.0 mm/s = 23.5 mm; at 7.0 mm/s and 12.0 mm/s = 0.001/0.000 mm (perfect). Sharp threshold at exactly 6.0 mm/s == _IK_TARGET_EPS(1e-4 m) * 60 Hz, confirming the per-tick `jump < _IK_TARGET_EPS` reuse branch is the cause. /tmp/repro_drift2.py shows every tick returns the held sentinel (only `_dist` key) with err_before==err_after (no correction) while dist stays 0.000005. /tmp/repro_confirm.py: after 40 ticks of 5e-5 m drift, reported _dist['left']=4.6e-6 m but TRUE fingertip-to-target error = 0.002 m and the cached q_best == measured config. The error grows linearly with elapsed time, not bounded by a transient. The existing smooth-tracking gate uses amp=0.25*sin (several mm/tick), so it never enters this sub-eps regime and misses the bug.
- **Fix:** Stop using the rolling per-tick target as the reuse reference. Add a separate cache field (e.g. `solve_target`) that is snapshotted ONLY when a real solve/polish refreshes `q_best` (the cold, tracking, and pump-cold branches), and compute the steady-reuse decision's `jump` against `cache["solve_target"]` rather than the per-tick-overwritten `cache["targets"]`. Then accumulated creep eventually exceeds `_IK_TARGET_EPS` (after ~2 ticks at 3 mm/s), falling through to the tracking branch (line 855), which already tracks slow motion to ~0 mm (verified). Equivalently/additionally, do not early-return the held sentinel (906-907) on the stale `cache["dist"]` while the LIVE fingertip-to-current-target error exceeds IK_POS_TOL: recompute the per-arm error from the current `pos[a]` and FK, and always run the per-arm Cartesian polish (912-935) when that live error is non-trivial, so the polish can trim the lag even on a reuse tick. The first fix is preferred (cheaper, addresses the root cause); keep the cheap-hold optimization only for a genuinely static target (cumulative drift, not per-tick drift, below the eps). Also ensure the returned `_dist` reflects the CURRENT target error, not a frozen stale residual, so callers/HUD aren't lied to.

### BUG 02 — [HIGH] A non-finite (NaN/Inf) target_pose crashes the whole controller (uncaught SystemExit/LinAlgError out of the 60 Hz tick)
- **File:** `ros2_ws/src/m1_control/m1_control/controller_node.py` lines 149-159, 184-191, 315-325 (category: crash, finder: controller-node, verifier confidence: high)
- **Problem:** _on_target() stores the incoming PoseStamped position verbatim (np.array([p.x, p.y, p.z])) with no finite-value validation, and _control_tick() calls self.reach.solve_step(...) with no guard. If any operator/external publisher sends a target_pose whose position contains NaN or Inf (e.g. a divide-by-zero in a teleop/quest delta calc, an uninitialized field, a bad headset matrix), the value flows into the Drake IK. A NaN makes Drake's constraint constructor abort via DRAKE_DEMAND, which pydrake raises as SystemExit; an Inf makes the DLS Cartesian polish raise numpy.linalg.LinAlgError ('SVD did not converge'). Neither is caught in _control_tick, and main()'s try/except only catches KeyboardInterrupt, so SystemExit propagates out of rclpy.spin, runs the finally (destroy_node/shutdown), and the controller process exits. A single malformed message thus kills the central control brain.
- **Evidence:** Ran the real node (rclpy.init, no DDS spin): seeded full /joint_states, published target_pose left=(nan,0.2,0.9), called _control_tick() -> raised SystemExit at drake constraint.h:72 ('!lower_bound_.array().isNaN().any()'), traceback through solve_step->_pump_cold->dik.solve->AddPositionCost. With Inf instead -> raised numpy LinAlgError 'SVD did not converge' from the DLS polish. Separately verified a timer callback raising SystemExit propagates out of rclpy.spin (returns after the offending tick), and main()'s 'except KeyboardInterrupt' does not catch SystemExit -> process exits. A normal reachable target (0.30,0.20,0.95) produces an all-finite command, confirming the crash is specific to non-finite input.
- **Fix:** Two-layer fix. (1) Validate at the ingress in controller_node._on_target (around line 149-159): after building new = np.array([p.x,p.y,p.z]), do `if not np.all(np.isfinite(new)): self.get_logger().warning(f"dropping non-finite {arm} target_pose"); return` so the bad value never updates self.targets[arm]. (2) Defense-in-depth in _control_tick around the solve_step call (line 187): wrap it in try/except. Because Drake surfaces a DRAKE_DEMAND failure as SystemExit (a BaseException, not Exception), the except must catch BaseException for the solve (e.g. `try: result = self.reach.solve_step(...) except BaseException as exc: self.get_logger().error(f"solve_step failed, holding last command: {exc}")` and skip updating pos_cmd that tick), so a single bad solve can never terminate the 60 Hz loop; the previous pos_cmd is held. Additionally, harden the operator publishers (web_node/quest_node _set_target) to reject non-finite x/y/z before publishing — the unbounded-z clamp (-inf,inf) cannot sanitize NaN/Inf — e.g. replace any non-finite component with the previous target value, and consider parsing web POSTs with json.loads(..., parse_constant=...) to reject Infinity/NaN. Lowered severity from critical to high: it requires a non-finite target value to reach the node (operator JSON with Infinity/NaN, or upstream arithmetic producing NaN), not a steady-state condition, but it is a real one-message kill of the central brain so it is more than medium.

### BUG 03 — [HIGH] on_activate seeds commands from never-populated pos_states_ (== 0), so the arm snaps to all-zeros instead of holding its measured pose on activation
- **File:** `ros2_ws/src/m1_hardware/src/m1_system_interface.cpp` lines 684-692 (with 629-651, 741-796) (category: correctness, finder: cpp-hw, verifier confidence: high)
- **Problem:** on_activate's stated purpose is 'Seed command interfaces from current state so we hold position on activate' (line 681). It does `pos_commands_[si] = pos_states_[si]` and writes that into pos_cmd_ifaces_[k]. But pos_states_ is ONLY ever populated from the live bus inside read() (line 760) or the mock echo (line 779). Neither on_init (line 561 sets it to 0.0), on_configure (which does openarm_->refresh_all()/recv_all() that updates the internal openarm Motor objects but NOT pos_states_), nor on_activate (which does enable_all()/recv_all(), same) ever copies motor.get_position() into pos_states_ before the seeding runs. read() is the only writer and it does not execute until the active control loop, i.e. AFTER on_activate. Therefore at activation pos_states_ is all 0.0, so the command interfaces are seeded to 0.0, not the measured joint positions. On real hardware, activating the controller while the arms are at a non-zero pose commands every joint to position 0 at full configured kp — a violent lurch to the folded all-zeros posture (which AGENTS.md notes folds the low arms into the drivebase). The 'hold position on activate' safety intent is silently defeated.
- **Evidence:** grep of all pos_states_ writers in the file: line 561 (assign 0.0), line 760 (read() from bus), line 779 (read() mock echo) — nowhere else. on_configure (629-651) calls openarm_->refresh_all()+recv_all() which decode feedback into the openarm Motor objects (DMCANDevice callback_mode_ defaults to STATE, dm_motor_device.cpp:27), but the plugin never copies motor.get_position() into pos_states_ there or in on_activate. So line 687 (pos_commands_[si] = pos_states_[si]) reads pos_states_=0.0. The exported command interface is set to 0.0 (line 690) and, until a brain command arrives, the forward_command_controller forwards it — commanding the motors to 0 rad at full kp.
- **Fix:** In on_configure (after the bus_ok_ refresh_all/recv_all at lines 641-643) AND/OR at the start of on_activate's seeding (before line 684), copy the freshly-decoded motor feedback into pos_states_ for each commanded motor using read()'s exact math, then seed commands from it. Concretely, when bus_ok_ && openarm_ and device indices are resolved: `auto & arm = openarm_->get_arm(); for (const auto & m : motors_) { if (m.device_index < 0) continue; const auto motor = arm.get_motor(m.device_index); pos_states_[m.state_index] = m.direction * motor.get_position() + m.offset; }` This must run BEFORE the `pos_commands_[si] = pos_states_[si]` loop so the seed reflects the measured pose. device_index is already resolved by try_open_bus()->resolve_device_indices() called in on_configure, so the get_motor(m.device_index) path is valid there. Equivalently, factor read()'s feedback-copy into a helper and call it once at the end of on_configure/on_activate. (Note: in the no-bus/mock case pos_states_ legitimately stays 0, which is fine — the danger is only the live-bus path.)

### BUG 04 — [HIGH] Controller init gate requires base steering joints, so arms-only hardware bring-up (use_base:=false) never initializes and never moves
- **File:** `ros2_ws/src/m1_control/m1_control/controller_node.py` lines 143-147 (and 68-76, 176-177) (category: contract, finder: contract-invariants, verifier confidence: high)
- **Problem:** M1Controller._on_joint_states gates initialization on `all(j in self.q_meas for j in POSITION_JOINTS)`. POSITION_JOINTS (line 68) = STEER_JOINTS + [lift] + left arm + right arm + fingers, i.e. it INCLUDES the 4 base steering joints (fl/fr/rr/rl_steering_joint). On the real-hardware seam, /joint_states for the upper body comes from joint_state_broadcaster, which (per m1.ros2_control.xacro / m1_controllers.yaml) publishes ONLY the 19 upper-body joints; the 4 steering joints are published ONLY by ranger_state_shim, which runs ONLY when use_base:=true. So when the stack is launched with the base off (hardware.launch.py default use_base:=false, and the documented arms-only and mock flows), the 4 steering joints never appear in q_meas, `_initialized` is never set True, and `_control_tick` returns early at line 176-177 forever. The brain therefore never publishes /m1/joint_command and the arms never move, even though all arm+lift feedback is present. The sim path (bringup.launch.py) is unaffected because Isaac publishes all 27 joints, which is why offline/sim validation misses it.
- **Evidence:** Built POSITION_JOINTS from the same modules the node imports: it contains ['fl_steering_joint','fr_steering_joint','rr_steering_joint','rl_steering_joint']. The joint_state_broadcaster system (m1.ros2_control.xacro) contains only the 19 upper-body joints (no steering). Reproduced the init gate with a hardware-style q_meas of the 19 upper-body joints: `all(j in q_meas for j in POSITION_JOINTS)` returns False, blocked on exactly the 4 steering joints; with the full 27-joint sim feedback it returns True. Keep the `for j in POSITION_JOINTS: self.pos_cmd[j] = self.q_meas[j]` seeding (lines 144-145) but make it tolerant of missing steer joints, e.g. `self.pos_cmd[j] = self.q_meas.get(j, 0.0)` for j in POSITION_JOINTS, so steer entries default to 0 until the swerve solve fills them. This lets the controller initialize and actuate the arms/lift on the arms-only hardware/mock path while remaining correct when the base is present.

### BUG 05 — [HIGH] Default mock hardware.launch.py (use_base:=false) never feeds the 4 base steering joints into /joint_states, so the brain never initializes and never publishes /m1/joint_command

### BUG 06 — [HIGH] NaN/inf in a target_pose or /joint_states crashes the controller's control loop (node dies)
- **File:** `ros2_ws/src/m1_control/m1_control/kinematics.py` lines 806-935 (solve_step); abort at 1072/1075, LinAlgError at 604 (category: crash, finder: edge-fuzz, verifier confidence: high)
- **Problem:** ReachController.solve_step does no finiteness validation of either the target point or q_meas. A NaN target coordinate flows into Drake's AddPositionCost / AddBoundingBoxConstraint and trips Drake's internal assertion ('!lower_bound_.array().isNaN().any()'), which pydrake surfaces as a Python SystemExit. An inf coordinate (or NaN in q_meas) instead reaches numpy.linalg.svd in _dls and raises LinAlgError('SVD did not converge'). controller_node._control_tick (line 187) calls solve_step inside a ROS timer callback with NO try/except, and main() only catches KeyboardInterrupt, so the exception propagates out of rclpy.spin and the controller node terminates. The arms then freeze at the last /m1/joint_command (no further commands published). The NaN case is especially severe: SystemExit is NOT a subclass of Exception, so even a defensive 'except Exception' wrapper would not catch it. The bad value can arrive from any publisher: _on_target (line 149-159) copies PoseStamped.position.{x,y,z} verbatim with no finite check, and _on_joint_states (line 140-142) does float(pos) with no finite check.
- **Evidence:** Repro A (NaN target): solve_step(q0, {'left':[nan,0.2,0.9]}) -> 'Failure ... condition !lower_bound_.array().isNaN().any() failed' raised as SystemExit; 'ONLY caught by except BaseException (SystemExit escapes except Exception)'. Repro B (inf target): solve_step(q,{'left':[inf,0.2,0.9]}) -> numpy.linalg.LinAlgError: SVD did not converge. Both propagate uncaught through controller_node._control_tick -> rclpy.spin -> node destroyed. Confirmed by running /tmp/repro_crash.py and /tmp/repro_crash2.py.
- **Fix:** Two layers. (1) Validate/sanitize inputs at the boundary: in solve_step (or _normalize_targets) drop any arm whose target is not all-finite (np.isfinite(pos[a]).all()), and replace non-finite entries of q_meas_vec with the cached/last value or 0.0 — so a single bad sample is skipped rather than fatal. Even cheaper and equally effective: reject non-finite at the publishers — in controller_node._on_target ignore the message if not np.isfinite([p.x,p.y,p.z]).all(), and in _on_joint_states skip any non-finite pos. (2) Defense in depth in the control loop: wrap the solve_step call in _control_tick in try/except BaseException (BaseException is required because Drake's NaN failure surfaces as SystemExit, which a plain `except Exception` will NOT catch); on failure log a throttled warning and hold the previous command instead of letting the node die. Keeping the brain alive on a transient bad message is the safer behavior for a robot control loop.

### BUG 32 — [LOW] MIT/feedback quantizer rounds differently between Python and C++ codecs (FP op-order), so the config tool and live plugin emit different CAN bytes for the same value
- **File:** `ros2_ws/src/m1_can_tools/m1_can_tools/dm_protocol.py (and ros2_ws/src/m1_hardware/vendor/openarm_can/src/openarm/damiao_motor/dm_motor_control.cpp)` lines dm_protocol.py:91-104 ; dm_motor_control.cpp:205-210 (category: correctness, finder: cpp-codec-xcheck, verifier confidence: high)

- **Evidence:** Ran a dense cross-check: built a C++ replica of `double_to_uint`/`pack_mit_control_data` from dm_motor_control.cpp and compared byte-for-byte against dm_protocol.encode_mit/float_to_uint. - **Fix:** Make both codecs use an identical FP operation order so they are bit-for-bit equal. Simplest: change Python float_to_uint (dm_protocol.py:104) to mirror the C++ exactly — `return int((x - lo) / span * ((1 << bits) - 1))` — so both divide-then-multiply. (Note this aligns the Python codec to whatever the vendored upstream openarm_can C++ does, which is the codec that actually drives motors.) Then add a cross-codec byte-equality test to the gated m1_can_tools suite: generate a C++ reference table (or port double_to_uint into the test) and assert encode_mit produces identical bytes over a dense parameter sweep including boundary-rational values like x = lo + k/(2^bits-1)*span, which are the only inputs that diverge. Optionally switch both to round-half (`int(... + 0.5)`) for nearest-integer quantization, but only if that matches the Damiao reference firmware — verify against the vendor spec before changing rounding, since the C++ truncation is the upstream behavior.

I traced on_xr_frame (lines 379-548) and confirmed `self.odom.reset()` is at line 390 — the FIRST thing inside the lock when `place` is true. The only code before it (lines 382-389) is plain dict `.get()` calls on already-parsed JSON, which cannot raise. Every reachable path to a 400 (do_POST lines 958-959 catch) requires an exception LATER — in the per-arm loop, _build_viz FK/serialization (line 547), etc. — i.e. AFTER the reset already ran.

I confirmed this by running a repro (/usr/bin/python3, ROS sourced, real M1QuestNode): set odom to a drifted pose (5,7,1.2), monkeypatched _build_viz to raise (simulating the realistic 400 case), posted a place frame, and observed odom = (0.0, 0.0, 0.0), theta 0.0 — the reset DID happen before the exception. So when the client clears pendingPlace on that viz-less 400, the SERVER ODOM IS ALREADY ZERO; the next successful POST returns base.p≈[0,0,0] and robotBase sits at the anchor. The recenter is NOT lost; clearing pendingPlace is in fact correct there.

The only true residual is purely cosmetic and marginal: vizPrev is not nulled on the viz-less response, so on resuming (pendingPlace false) there could be ONE ~20-40 ms interpolation frame between a stale pre-place base and the new zeroed base — bounded because the new base is near-origin/anchor. This needs an unrelated server-side exception on a place frame (the codebase always sends well-formed JSON), affects only the live-only viz/preview path, and never touches the 60 Hz arm/base commands. It is not a wrong-result, crash, contract-violation, race, leak, or bad-hardware-command defect.
6. **transport-bus** — SerialTransport.recv loses CAN frames on every short read (no rx byte-buffer)
   - Why not a bug: The candidate's central premise is wrong. It claims "every read is non-blocking because the port is opened at timeout=0.0 (line 161), so short reads are the NORMAL case and frames are permanently lost." But recv() at transport.py:187 overwrites `ser.timeout = timeout` on EVERY call, so the construction-time 0.0 is never used for reads. The actual callers (motor_bus.telemetry line 147/155, scan line 177/198) pass timeout=0.01 (10 ms).\n\nI read pyserial 3.5's actual read() implementation (/usr/lib/python3/dist-packages/serial/serialposix.py:553-602): read(N) is a LOOP that select()s and accumulates bytes until len(read)>=N OR the timeout expires. So a frame split across multiple OS reads within the timeout window is reassembled by pyserial itself — the recv() wrapper only sees the final, complete result. The candidate's repro stubbed read(n) to return one arrival-chunk per call, which deletes pyserial's internal accumulation loop — i.e. it stubbed out the exact mechanism that makes the code correct, so the repro is not faithful to pyserial.\n\nI confirmed against a real pty + real pyserial: (Scenario B) a frame written as 10 bytes, 2 ms gap, then 6 bytes, with timeout=0.01, is correctly reassembled and recv() returns (0x12, data) — NOT None. (Scenario 2) three aligned back-to-back frames all decode correctly. At 921600 baud a 16-byte frame transmits in ~174 us, which fits 58x inside the 10 ms read timeout, so the 'short read every poll' scenario is physically implausible. The construction timeout=0.0 IS overridden (verified: setter takes effect, read(16) blocks ~3 ms and returns all 16 bytes).\n\nThere is a genuine but much narrower robustness gap: there is no rx byte-buffer and no 0x55/0xAA header resynchronization, so (a) a stream that starts byte-misaligned — a stray byte or half-frame present at connect — stays permanently misaligned and loses every frame (confirmed Scenario 3), and (b) an intra-frame gap exceeding the 10 ms read timeout discards the partial read and misaligns the stream (Scenario 1). But these are abnormal/degraded conditions, not the normal path, and this is a bench/maintenance-only serial dongle path (m1_hwconfig transport:=serial), explicitly documented as a bring-up convenience while the production run path is the C++ SocketCAN plugin. The high-severity 'frames lost on every short read' defect as described does not manifest on the real path.
7. **hwconfig** — api_limits IndexErrors on a malformed (scalar or short) pos payload before the lock; returned to client as a generic 400 rather than a clear validation error
   - Why not a bug: I read hwconfig_node.py lines 205-235 (api_limits) and 383-394 (do_POST) in full and ran the candidate's repro under /usr/bin/python3 with PYTHONPATH=ros2_ws/src/m1_can_tools. A well-formed pos=[-0.5,0.5] returns (200, {'ok':True}). So the technical claim is accurate.

However this is NOT a genuine defect by the task's bar. The exception is fully caught by do_POST's try/except (lines 393-394) and converted into a clean HTTP 400 with `_send` writing a proper status line + body — the ThreadingHTTPServer does not crash, the connection is not wedged, no thread dies, no resource leaks. Crucially, the malformed value is rejected: it never reaches `info['soft_limits']`, the `_write_limits_yaml()` writer, or any motor/CAN command. There is no wrong numerical result, no contract violation, no race, no incorrect hardware command. The ONLY consequence is a less-descriptive error string ("'float' object is not subscriptable" instead of "pos must be [lo,hi]"). The finder itself concedes "it is a robustness/UX defect, not a server crash." Note other API methods in the same file (api_assign, api_map) likewise rely on int()/get() raising into the same outer handler — this is the established, intentional error-handling pattern for this maintenance tool's config page, not a one-off oversight.
8. **cpp-hw** — write() forwards the command interface to the motors with no NaN guard; ros2_control resets command interfaces to NaN on controller activation, so a NaN setpoint packs to q=0x0000 = -pMax (full-limit runaway)
   - Why not a bug: The candidate's NaN-packing math is correct (I compiled openarm_can's exact limit_min_max/double_to_uint: double_to_uint(NaN,-12.5,12.5,16) returns 0x0000, which decodes to -pMax = -12.5 rad — a dangerous full-limit command IF it reached write()). (2) update() begins with `if (std::all_of(joint_commands_.data, isnan)) return OK;` — confirmed by the disasm NaN-scan loop at 0x3cf60 whose all-NaN path branches to the early return at 0x3d238 (sets return OK, skips the write loop at 0x3d020). (3) The hardware's own on_export_command_interfaces (line 622, ci->set_value(pos_commands_[si]=0.0)) and on_activate (line 690, set_value(pos_states_)) seed pos_cmd_ifaces_ to FINITE values, so they are never left at the Handle default NaN. 9. **concurrency** — Slow bus scan holds _lock across blocking recv(), starving the jog dead-man safety disable
   - Why not a bug: The candidate's mechanical claim is true and I reproduced it: api_scan (hwconfig_node.py:148-150) holds self._lock across MotorBus.scan (motor_bus.py:158-201), whose recv loop blocks until the bus goes quiet, and tick_deadman (hwconfig_node.py:313-328) needs the SAME lock to fire its safety-disable. My first repro (an adversarial transport returning 40 frames each blocking 50ms) showed api_scan held _lock ~2.0s and delayed the deadman disable ~2.0s. BUT that scenario is not reachable in the documented operating mode, so it is not a real defect:

1. Maintenance-mode scan is NON-ENERGIZING BY DESIGN (motor_bus.py:158-201 docstring; test_scan_is_non_energizing). Unenabled DM motors do not stream feedback — they reply once per refresh poll, then the bus is quiet. SocketCanTransport.recv always passes timeout=0.01 (transport.py:116-118), so each recv is bounded to 10ms and the scan loop ends on the first None. My realistic repro (17 motors present, each replying once) measured the scan holding the lock ~10ms — and ~10ms even scanning 50 ids — not seconds. Bus ownership is exclusive (maintenance XOR run), so there is no other continuous-traffic source. The 'continuously-chattering bus → seconds-long scan' premise contradicts the mode the tool runs in.

2. The deadman-vs-scan contention is essentially impossible from the real client (hwconfig.html). Jog is a press-and-hold control (mousedown/mouseup/mouseleave -> stopJog) refreshed every 100ms via setInterval(jogTick,100); JOG_HOLD is 500ms. To click 'Scan bus' on a single pointer you release/leave the jog control (firing stopJog, jogging=false) — so you cannot simultaneously hold a stale jog and issue a scan. And the dead-man's real purpose is to catch a DEAD/DISCONNECTED browser, which is exactly the case where no scan POST is in flight, so there is no lock contention then.

3. Worst realistic delay is ~10ms (one extra 50ms deadman tick at most), far under the 500ms JOG_HOLD tolerance. poll_telemetry shares the executor thread with tick_deadman (sequential, not contending) and uses timeout=0.0, so it is not a factor.

So this is at most a minor code smell (lock held across a bounded blocking I/O call); the dead-man safety guarantee holds in all realistic conditions. The 'confirmed_by_running' in the candidate demonstrates only the abstract lock mechanics via an injected adversarial transport, not a defect reachable in normal operation.
